# 実リポジトリへの適用ガイド

検証済みの single-agent closed loop を、**実プロジェクトのリポジトリ**で回すための手順。
この評価リポジトリ（`loop-arch-evaluation`）は砂場兼リファレンスで、実適用は対象リポジトリ側で行う。

## 3 つの構成要素

1. **理論**: <https://hidekingerz.github.io/catch-all-favorite/content/research/single-agent-loop/>
2. **ポータブル雛形**: [`templates/single-agent-loop/`](../templates/single-agent-loop/)
   — 別リポジトリに**フォルダごとコピー**して使える（プレースホルダ入り）。転用手順は雛形内の
   [`README.md`](../templates/single-agent-loop/README.md) を参照。
3. **検証の裏付け**: [`docs/verification-report.md`](./verification-report.md)
   — 収束 / 安全停止 / 依存解決 / 人間判断 / リファクタ / バグ修正 / maker-checker / 機能ビルド /
   分解ヒントなし / 長期収束（MEMORY 背骨）の全 10 系統で機構を確認済み。

## 手順（全体像）

1. **対象リポジトリにスコープしたセッション/環境を立てる**（Claude Code on the web/app）。
   依存インストールと実テストが通るネットワークポリシー/セットアップにする。
   設定方法: <https://code.claude.com/docs/en/claude-code-on-the-web>
2. **ポータブル雛形をフォルダごと持ち込む**: `templates/single-agent-loop/` を対象リポの `loop/` にコピー。
3. **プレースホルダ（`{{...}}`）を対象向けに記入**（最重要は VERIFY と完了条件）:
   - `run.sh` の `VERIFY_CMD` = 対象リポの実コマンド（例: `npm run typecheck && npm run lint && npm test`）
   - `ARCHITECTURE.md` = 実スタック・主要ディレクトリ・実コマンド
   - `RULES.md` = 対象固有の禁止事項（本番設定を触らない・依存追加禁止 等）＋共通ガードレール
   - `VISION.md` = **最初の 1 タスク**と検証可能な「完了の定義」
4. **最初の標的は小さく・完了がバイナリに測れるもの**（再現テスト付きのバグ修正、テストで表せる小機能）。
   大きい/曖昧な課題は 1 周目に選ばない。
5. **まず 1 周だけ回す**: `MAX_ITER=1 ./run.sh`（CLI ランナー）か、Claude Code の `/loop`、
   あるいは下のキックオフ指示でセッション自身に 1 周させる。挙動を見てから周回を増やす。

## 新セッションに貼るキックオフ指示（コピペ）

対象リポジトリにスコープした新セッションの**最初のメッセージ**に、以下をそのまま貼る。
`{{...}}` は必要なら埋める（空ならセッション側が選ぶ）。

```text
あなたはこのリポジトリの実コードに、検証済みの「single-agent closed loop（閉ループ）」を適用します。
いきなり大きく作らず、1周＝1タスク＋実テストでの検証＋コミット、を小さく回します。

# フェーズ0: 偵察
1. package.json の scripts を読み、実際の「型チェック / lint / テスト / ビルド」コマンドを特定する。
   依存を入れて（npm ci 等）、それらが実際に走ることを確認する。
2. プロジェクト構成と open issue をざっと把握する。
3. 使う VERIFY コマンドの確定結果と、最初の標的タスク候補を報告する。

# フェーズ1: ループ雛形を loop/ に用意
- 可能なら検証済みポータブル雛形をコピーする:
  hidekingerz/loop-arch-evaluation の templates/single-agent-loop/ を loop/ にコピー
  （同セッションのスコープに無ければ、下記の役割で loop/ 配下に同等のファイルを新規作成する）。
- loop/ の中身をこのリポ向けに記入する:
  - run.sh の VERIFY_CMD = フェーズ0で確定した実コマンド。
  - ARCHITECTURE.md = 実スタック・主要ディレクトリ・実コマンド。
  - RULES.md（ガードレール、最優先）:
    * テストファイルを変更/削除/スキップしない（テストは仕様）。
    * テスト入力に合わせたハードコード/対症療法で通さない（一般化する実ロジックで直す）。
    * 依存追加/更新・ビルド/ツール設定の変更・本番設定の変更をしない。
    * 公開API（props/引数/戻り値の形）を勝手に変えない。
    * VERIFY を通っていない変更をコミットしない。1周=1タスク。秘密情報をコード/ログに書かない。
  - VISION.md = 最初の1タスクと、その「完了の定義（検証可能な箇条書き）」。
  - MEMORY.md = Done/Open/Notes の空欄。Open に開始状態を1行記録。
  - LOOP_PROMPT.md = 下記5段階。VERIFY=実コマンド、停止条件=DoD全達成＋全緑で「LOOP_DONE」。

# フェーズ2: 最初の標的タスクを1つ選ぶ（小さく・完了がバイナリに測れる）
- 候補: {{最初に狙う issue 番号 or タスク。空なら open issue から境界が明確なものを1つ選ぶ}}
- 再現/受け入れテストが無ければ、先に「失敗するテスト」を書く（これが仕様）。
  意図どおりの理由で赤くなることを確認してから実装に入る。
- 設計判断が要る/受け入れ条件を言語化しにくい課題は1周目に選ばない。

# フェーズ3: 5段階ループを1周だけ回す
1. DISCOVER: loop/ の4ファイルを読み、実テストを走らせ現状を把握。MEMORY.md の Open を確認。
2. PLAN: 1タスクだけ選び、根本原因の仮説/方針を1〜3行で宣言。
3. EXECUTE: 必要最小限の変更。RULES を厳守。
4. VERIFY: 実コマンド（型→lint→テスト）を実行。対象が緑かつ既存の緑を1つも壊していないこと。甘く採点しない。
5. ITERATE:
   - 失敗 → コミットしない。原因/観察を MEMORY.md の Open に具体的に記録して終了。
   - 成功 → フィーチャーブランチに実装＋MEMORY.md の Done 追記を1コミットにまとめる。

# 約束
- フィーチャーブランチで作業。main へ直接 push しない。PR 作成は明示的に頼まれた時だけ。
- 1周ごとに、やったこと・実VERIFY結果・コミット・MEMORYの次の申し送りを報告し、続行可否を確認する。
- まずは1周だけ。挙動を確認してから周回を増やす。

最初にフェーズ0（偵察）を実行し、確定したVERIFYコマンドと最初の標的タスク候補を報告してください。
```

## メモ

- 上のキックオフは**ポータブル雛形のコピーを起点**にしつつ、雛形を持ち込めない場合は同等ファイルを
  新規作成するフォールバックも含めた自己完結型。
- maker/checker 分離（VERIFY を誠実にする 2 役）は、慣れてから・重要変更で導入するとよい
  （検証は `docs/verification-report.md` の G を参照）。
