# RULES — エージェントが絶対にやってはいけないこと

> VERIFY の品質ゲートと並ぶ「ガードレール」です。PLAN / EXECUTE でこのルールを最優先で守ります。

## 禁止事項（Hard rules）

- テストファイル（`src/**/*.test.ts`, `src/**/*.test.tsx`）を**変更・削除・スキップしない**
  （`it.skip` / `it.only` / コメントアウト含む）。テストは仕様であり不可侵。
- **対症療法をしない**。テストの入力値（例: 特定の id・特定のクエリ文字列）に合わせた
  ハードコード分岐や `if (count === ...)` のような特別扱いで症状だけ消すのは禁止。
  **必ず根本原因を直す**（クロージャ・不変更新・依存配列など、実際の不具合を修正する）。
- **既存で緑のテストが守る振る舞いを壊さない**。修正後に回帰がないこと。
- 公開エクスポートのシグネチャ（props・引数・戻り値の形）を変えない。
- 依存パッケージの追加・削除・バージョン変更をしない。ビルド/ツール設定
  （`tsconfig.json` / `eslint.config.js` / `vitest.config.ts` / `package.json`）を変更しない。
- VERIFY を通っていない変更をコミットしない。1周で複数ユニットにまたがる修正をしない。
- `templates/single-agent-loop/` を変更しない。

## 迷ったときの原則

- 「なぜこのテストが落ちるのか」を1〜2文で**原因仮説**として言語化してから直す。
- 仮説が外れて直らない場合は、握りつぶさずに `MEMORY.md` の "Open" に観察を残して次周へ。
- 1周 = 1ユニット。スコープを広げない。
